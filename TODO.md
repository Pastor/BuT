## 2. Доступ до конкретного бита

Необходимо добавить операцию получения/изменения конкретного бита переменной не математического типа. Операция точка `.`.
Если передается имя переменной и после нее идет точка с числом - `.34` то надо считать что идет обращение к 35 
(начиная с 0, младшего адреса) биту переменной. Система типов должна делать проверку на уровне семантического анализа, что 
при обращении к биту нет выхода за границы типа. При кодогенерации используются языковые конструкции доступа к биту данных.

Примеры:
```
let a: [8: bit] = 0;

a.5 = true; // Выставление 6 бита в true
```

```
let b: [128: u8];

b.250 = false; // Обращение к 251 биту области памяти
```

```
let c: [b: bit]

c.64 = true; // Ошибка, выход за границы типа
```

## 5. Обработка сложных моделей

Проанализируй проект. Посмотри учтена ли сложная компоновка моделей, например ((A + B) | C) + D. При необходимости внеси изменения для обеспечения этого требования. Рассмотри процесс генерации кода. При генерации кода, сейчас каждая модель генерится отдельно и добавляется в код, без учета компоновки `behavior`. Необходимо скорректировать генерацию кода для всех поддерживаемых языков так
чтобы в заголовочном файле присутствовала одна функция инициализации (с передачей в качестве параметра структуру (указатель на структуру) модели для инициализации), функция одного шага модели, функция остановки модели и структура описывающая эту модель. 
Все функции других моделей не должны быть видны в заголовочном файле и присутствовать только в реализации (это необходимо для инкапсуляции реализации модели). Это необходимо сделать для всех генераторов языков для которых это возможно. В конечном итоге вызывающая сторона должна
Оперировать только одним набором функций. Если поведение модели имеет сложное описание (сложную компоновку) то в качестве имен функций инициализации, шага и остановки (завершения) необходимо использовать имя файла. Пример: `engine.but` в нем создается две модели
`Consumer` и `Producer` и имеют сложную компоновку. При генерации для языка си, в заголовочном файле будут присутствовать следующие объекты: `struct Engine`, `void Engine_init(struct Engine *engine)`, `void Engine_step(struct Engine *engine)`, `void Engine_destroy(struct Engine *engine)`, `void Engine_is_done(struct Engine *engine)`. 
Внутри же файла реализации будут, также, присутствовать функции, в соответствии с определением моделей, `struct Consumer`, `static void Consumer_init(struct Consumer *consumer)`, `static void Consumer_step(struct Consumer *consumer)`, `static void Consumer_destroy(struct Consumer *consumer)`, `static void Consumer_is_done(struct Consumer *consumer)` и т.д.
Если присутствуют `extern` функции или переменные они должны присутствовать в заголовочном файле. Соответственно в функции `void Engine_init(struct Engine *engine)` будут инициализироваться все модели входящие в основную, в функции `void Engine_step(struct Engine *engine)` будут вызываться входящие в нее функции моделей исходя из их компоновки и т.д.
Для языков `ST` и ассемблеров следует не создавать функции для вызова, а разворачивать их в коде.
